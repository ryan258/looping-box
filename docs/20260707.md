# Project Review — 2026-07-07

Full-repo review: `phase1`, `worker`, `supervisor`, `review`, `model`,
`action_policy`, `schema`, configs, demos, docs, tests. Test suite: 41/41
passing. Items already documented as by-design in `REVIEW.md` (substring
keyword matching, one-blocked-item-stalls-the-batch, `processed_files`
growth, loud failure on malformed SOP) are not repeated here.

## High

### H1. Symlink inside `inbox/` can exfiltrate any in-root file (verified)

`_iter_input_files` (`src/looping_box/phase1.py:196-217`) rejects symlinks
whose real target escapes the repo root, but allows targets *inside* root.
The extension check uses the symlink's own name, not the target's. So
`ln -s ../.env inbox/leak.md` passes both checks, and the `.env` content —
including `OPENROUTER_API_KEY` — is read, hashed, and excerpted into
`cache/deltas/*.json` and staging review payloads. If `MODEL_CONTEXT_BUILDER`
is configured, the excerpt is also sent to OpenRouter.

Verified by repro: the delta's `changes[0].excerpt` contained the full key
line. (The gate happened to trip only because the test value contained the
substring "secret" — content without a gate keyword flows straight through
to workers and the model.)

**Fix (lazy):** skip symlinks entirely in `_iter_input_files`
(`if path.is_symlink(): continue`), or require the resolved target to be
under `input_dir` rather than under root. Add a test mirroring
`test_symlink_escaping_root_is_not_ingested` for the in-root case.

### H2. Recovery instruction for resource-limit blocks points at the wrong command

`status_summary` (`src/looping_box/supervisor.py:78-84`) tells the operator
to fix `config/super_loop.json` "then run ./startday.sh". `startday.sh` only
runs phase1 — it never reruns the supervisor, so following that instruction
leaves the block in place and `--status` still reports blocked.
`docs/RECOVERY.md:96-99` has the correct command (`supervisor --once`).

**Fix:** change the next-step text to `python3 -m looping_box.supervisor
--once` (or make `startday.sh` run phase1 + supervisor, which would also
match the README's framing of it as "the loop").

## Medium

### M1. Action classes are informational only — `forbidden`/`blocked` items are approvable

`classify_reasons` stamps an `action_class` on review payloads, and the
config defines a `forbidden` tier, but nothing enforces it:
`record_review` (`src/looping_box/review.py:48-83`) approves a
`forbidden`-class item exactly like a `review_required` one. The taxonomy
implies an escalation ladder that doesn't exist.

**Fix:** in `record_review`, refuse `decision == "approved"` when the
payload's `action_class` is `forbidden` (and arguably `blocked` without an
explicit override flag). One `if`, one test.

### M2. `worker_timeout` blocks can silently self-clear, contradicting the stated guarantee

README (`Guarantees`, lines 79-81) and `docs/RECOVERY.md:87-90` say
resource-limit blocks "never silently self-clear". That holds for
`file_count_limit` and `payload_size_limit` (deterministic re-trip), but the
runtime check is wall-clock: a transiently slow run (slow disk, slow model
response) blocks, the rollback queues the identical work, and the next
`--once` may finish under budget and clear the block with no operator
action. Nothing in `_run_supervisor_locked` gates on
`recovery.operator_action_required`, so the retry happens on any rerun.

Related config mismatch: the model transport timeout is 60s
(`src/looping_box/model.py:63`) while `max_worker_runtime_seconds` defaults
to 30 — a slow model call always completes (and is paid for) and *then* gets
blocked post-hoc.

**Fix options:** (a) soften the README/RECOVERY claim to "deterministic
limits never self-clear; timeout blocks retry the identical work"; or
(b) make the supervisor refuse to run while `operator_action_required` is
set until the config hash changes. Also consider deriving the HTTP timeout
from `max_worker_runtime_seconds`.

### M3. Rollback restores worker state but not worker artifacts

`_blocked_with_rollback` (`src/looping_box/supervisor.py:239-252`) restores
only `cache/workers/<id>/state.json`. Artifacts written by the blocked run —
`context_package.json/.md`, `draft.json/.md`, and `last_output.json` — stay
on disk. In particular `last_output.json` still records the worker's
pre-override status (e.g. `complete`) while `.world_state.json` says the run
was blocked and rolled back. Confusing during an audit, and the oversized
artifact from a `payload_size_limit` block persists.

**Fix:** snapshot/restore the whole `cache/workers/<id>/` directory, or at
minimum rewrite `last_output.json` with the overridden (blocked) output.

### M4. Content-hash dedup bypasses the boundary gate for identical content at a new path

In `run_phase1`, `_has_processed` runs *before* keyword matching
(`src/looping_box/phase1.py:65`), and once a review decision is recorded the
file's hash is registered in the global `processed_hashes`
(`phase1.py:101`, `:249-256`). Result: after approving **or rejecting**
`inbox/release.txt`, an identical copy dropped at `inbox/release-copy.txt`
is skipped as `already_processed` — no gate, no new review — even though its
path-specific review id was never decided. The CHANGELOG states suppression
is scoped to "the same source path, content hash, and review reason set";
the hash-only dedup makes it path-independent, and a *rejected* item's
content silently skipping elsewhere is the sharp edge.

**Fix:** don't register review-decided files in `processed_hashes` (keep the
per-path entry only), or check review keywords before the hash-dedup
short-circuit. Add a test.

## Low

### L1. `--once` is decorative

`supervisor.main` runs a pass whether or not `--once` is given
(`src/looping_box/supervisor.py:520-535`); only `--status` changes behavior.
Either make `--once` required for a run (safer against accidental
invocation) or drop it from the docs/help.

### L2. Duplicated helpers with divergent behavior

- `_read_json` is defined 3× (`phase1`, `supervisor`, `worker`, `review`).
- `_resolve_under_root` is duplicated in `action_policy.py:61-66`.
- `_decision_record_exists` is duplicated (`phase1.py:445`, `review.py:188`).
- `_read_pending_index` is duplicated **with different semantics**: phase1's
  tolerantly resets on unknown schema (`phase1.py:398-424`), review.py's
  raises (`review.py:160-173`). Whether a corrupt index is recoverable
  depends on which module touches it first.
- `supervisor`, `worker`, and `review` all import `_`-prefixed privates from
  `phase1` (`_rel`, `_resolve_under_root`, `_sha256_file`, `_utc_now`,
  `_write_json`) — phase1 is acting as the de-facto util module.

**Fix:** one small `_util.py` (or make those five helpers public in a
`common.py`); pick one pending-index read behavior (the tolerant one, since
review decisions are recoverable from `staging/approvals|rejections/`).

### L3. A single malformed delta permanently fails `context_builder`

Any unreadable JSON in `cache/deltas/` makes every `context_builder` run
return `failed` without consuming anything (`src/looping_box/worker.py:59-103`),
wedging the pipeline until the file is hand-deleted. Failing loud is fine;
consider quarantining the bad file (rename to `.bad`) so one corrupt delta
doesn't block all subsequent good ones, and mention the manual fix in
RECOVERY.md.

### L4. `context_builder` spends model calls on batches execution will refuse

The synthesis call runs whenever `items` is non-empty
(`src/looping_box/worker.py:127`), including when `blocked_inputs` makes the
package `status: "blocked"` — which `execution_engine` then refuses to
consume. Gate the model call on `status == "complete"` to avoid paying for
context nobody uses.

### L5. Model-verifier prompt injection

`_model_review_check` (`src/looping_box/review.py:124-149`) embeds the
untrusted source excerpt in the judge prompt. A file crafted to say
"reply PASS" can neutralize the model layer. The human approval and
deterministic checks remain the real gate, so this only weakens the optional
extra layer — but worth a comment, and the deterministic checks are thin
(schema + non-empty). Also trivial: the prompt is built before the
`is_enabled` early-return; swap the order.

### L6. Review payload's embedded verifier status never updates

Payloads are written with `verifier: {status: "pending"}`
(`phase1.py:340-344`) and stay that way forever; the actual result lives
only in `cache/verifiers/<id>.json`. Either update the payload on decision
or drop the embedded block and keep just `verifier.required`.

### L7. Lock hygiene

`run_supervisor`'s `finally` does `if lock_path.exists(): lock_path.unlink()`
(`supervisor.py:41-43`): (a) TOCTOU — use `unlink(missing_ok=True)`;
(b) if a run exceeds `stale_lock_seconds` and a second process steals the
lock, the first process deletes the second's lock on exit. A pid check
before unlink closes it; acceptable ceiling for a single-operator local
tool, but worth a `ponytail:` comment naming it.

### L8. `load_env` caches for process lifetime

`_env_loaded_for` (`src/looping_box/model.py:23,34-39`) means a long-lived
process never sees `.env` edits, and `os.environ.setdefault` means an edited
value never wins over the first-loaded one. Fine for one-shot CLI runs;
becomes a trap if daemon mode (on the roadmap) arrives. Leave a comment.

## Growth / performance

### G1. Unbounded run artifacts, re-read every cycle

- `run_phase1` writes a delta file **every run, even when nothing was
  scanned** (`phase1.py:125-152`), so `cache/deltas/` grows one file per
  invocation forever.
- The supervisor globs and JSON-parses *every* delta each pass
  (`_phase1_delta_paths` + `_changed_file_count`,
  `supervisor.py:333-356`) — O(total historical runs) work per cycle.
- `observed_deltas` in `.world_state.json` and `consumed_inputs` in worker
  state grow one entry per delta forever (`runs` is capped at 50; these
  aren't).

**Fix (lazy):** skip writing a delta when `scanned == 0` and nothing
changed, and have the supervisor move fully-observed deltas to
`cache/deltas/archive/` (then `observed_deltas`/`consumed_inputs` can stay
small). Extends the `processed_files` note already in REVIEW.md.

### G2. One model call per draft item

`_draft_item` (`worker.py:266-289`) makes N sequential API calls for N
items. Fine at inbox scale; batch into one prompt if cost/latency shows up.

## Enhancements

### E1. Packaging

No `pyproject.toml`; every command needs `PYTHONPATH=src`. A ~10-line
pyproject with `[project.scripts]` (`looping-box-phase1`, `looping-box-
supervisor`, `looping-box-review`) removes the incantation from README,
RECOVERY.md, and all three demo scripts.

### E2. Dead world-state fields

`leases`, `dirty`, and `registered_loops` are written and defaulted but
never read anywhere (`supervisor.py:186-188, 272, 302-313`). Delete them or
leave a `ponytail:` note saying what future phase they're reserved for —
right now they read as load-bearing and aren't.

### E3. Context markdown omits the model synthesis

`_context_markdown` (`worker.py:342-356`) doesn't include
`context["synthesis"]` when a model produced one — the human-readable
artifact is missing the only model-generated content.

### E4. Test gaps worth closing

- In-root symlink ingestion (H1) — only the escaping-root case is tested.
- Hash-dedup gate bypass at a second path (M4).
- Post-rollback consistency of `last_output.json` vs world state (M3).
- `forbidden` action class refusing approval (M1, once enforced).
